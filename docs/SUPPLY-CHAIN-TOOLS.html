TrustLink SCS — 공급망 보안 도구 확장 설계

기존 전략("엔진은 만들지 말고 통합한다") 위에서, 빈 곳을 채우는 OSS 도구 추천 + 통합 설계
한 줄 요약. 지금 플랫폼은 SBOM·VEX·서명을 zot referrer로 저장하지만 아무것도 검증·집행하지 않습니다. 전략 문서가 "P1 핵심·제품 정체성"으로 지목한 수용·검증 게이트가 여전히 미구현입니다. 따라서 추가 도구의 1순위는 새 기능을 늘리는 것보다 저장소를 "능동적 검증·집행 계층"으로 바꾸는 3종 세트입니다 — Trivy(실 CVE) · Cosign(서명/검증) · 검증 게이트(BFF 오케스트레이션).

3. 중심 설계 — 수용·검증 게이트

BFF("두뇌")가 push/승격(promote)/pull 시점에 검증을 오케스트레이션하고, 결과를 검증 리포트 referrer로 부착합니다. 위 도구들은 모두 이 게이트의 입력입니다.

점선 박스 = 통합 엔진(내부). 게이트는 결과를 검증 리포트 referrer로 부착해 불변·감사 가능하게 보관하고, FAIL은 승격/배포를 막습니다.
집행 지점

언제 게이트를 거나

  • push 직후(비동기) — 검증 리포트 생성·부착, 등급 표시
  • 승격(promote) 시 — dev→released 태그 이동 전 fail-closed
  • pull 정책(선택) — 미검증/FAIL 아티팩트 pull 차단(엄격 모드)
검증 규칙(초기)

무엇을 검사하나

  • 서명 유효 + 신뢰 키/identity (cosign)
  • SBOM 존재 + CycloneDX 스키마 + 바이너리 해시 일치
  • VEX의 PURL이 SBOM 컴포넌트와 매핑(대상 일치)
  • VEX 반영 후 잔존 Critical/High CVE ≤ 임계값

4. 데이터 모델 (zot referrers 확장)

전략 §7의 결합 모델에 provenance와 검증 리포트를 추가합니다. 모두 불변·누적·출처 보존 원칙 유지.

products/innoecm:1.2.3 # subject = 바이너리/이미지 ├─ source.cdx.json # SBOM (Sparrow/업로드) ├─ binary.cdx.json # SBOM (xSCAN/Syft 보조) ├─ vex.cdx.json # VEX (DT 발행, 최신 1건 반영) ├─ *.sigstore.bundle # 서명 (cosign) ← 검증 대상 ├─ provenance.slsa.json # NEW: 빌드 출처(SLSA/in-toto) └─ verification-report.json # NEW: 게이트 판정·등급·근거(불변)
